GitOps / CasC Bundle Delivery

All Jenkins configuration is managed as code in this Git repository — no clicking in the UI. Two parallel delivery paths exist: Path A handles the OC's own configuration bundle via the casc-retriever Quarkus service. Path B handles managed controller bundles via the git-sync sidecar and OC's bundleStorageService. Total propagation time from git push to live config: up to 3 minutes (1 min retriever + 2 min OC poll).

How ocBundleAutomaticVersion works

The casc-retriever replaces the version field in bundle.yaml with the current git commit SHA before writing to disk. OC treats this as a checksum — any push creates a unique version, triggering reload. You never manually bump the version number.

Networking & Ingress

The VPC spans 3 Availability Zones (us-east-1a/b/c). EKS nodes sit in private subnets with no public IPs. All inbound traffic enters through the ALB (Application Load Balancer), managed by the AWS Load Balancer Controller. TLS is terminated at the ALB using an ACM certificate (DNS-validated, auto-renewing). A single NAT Gateway in one AZ handles outbound traffic (cost trade-off: ~$32/mo, acceptable for a lab).

Why sticky sessions on the ALB?

SAML SSO login state is stored in-memory per OC pod. Without sticky sessions, the ALB could route the /finishLogin SAML callback to a different pod than the one that generated the AuthnRequest — causing a 403. Cookie-based stickiness (lb_cookie, 24-hour duration) pins each user's session to one pod.

Identity & SSO (Entra ID SAML 2.0)

Authentication uses Microsoft Entra ID (personal tenant) as the SAML 2.0 Identity Provider. The SSO Relay (Quarkus service, 2 replicas) acts as a SAML Service Provider intermediary — it handles the SAML AuthnRequest/Response flow, extracts the assertion, and passes the authenticated user to OC. Group membership is passed via the http://schemas.microsoft.com/ws/2008/06/identity/claims/groups claim as Entra group Object IDs.

Why SAML and not OIDC?

SAML 2.0 is preferred for CBCI active-active HA. In HA mode, OC must propagate authentication state across replicas — the SAML plugin handles this better than the OIDC plugin. It also mirrors the Mastercard production pattern and is more portable across IdPs.

forceAuthn: true — why it matters

Without this flag, Entra can return a cached SAML assertion from a previous session. If the cached assertion was issued more than maximumAuthenticationLifetime seconds ago, CBCI throws SAMLAuthnInstantException and the login fails. forceAuthn: true forces Entra to issue a fresh assertion every login.

EKS Cluster & Karpenter Node Pools

The cluster uses a hybrid node strategy: a fixed Managed Node Group (MNG) for critical system pods, and Karpenter for dynamic provisioning of controller and agent nodes. This gives stability for platform components (OC, SSO Relay, LB controller) while enabling cost-efficient, on-demand scaling for workloads. Three distinct NodePools enforce isolation via node labels and taints.

Why separate node pools for controllers and agents?

Agent pods are ephemeral and high-churn. Karpenter aggressively terminates idle spot nodes. Without isolation, a churning agent node could evict controller pods — breaking active builds and violating the HA SLA. The taint (workload=agents:NoSchedule) ensures only agent pods with the matching toleration land on agent nodes.

Operations Center — Internals

The OC pod runs 4 containers: one main Jenkins JVM, one casc-retriever Quarkus sidecar, one git-sync sidecar for controller bundles, and one cleanup init container that runs at startup. The OC is the CBCI control plane — it provisions managed controllers, distributes CasC bundles, and manages licenses. It is backed by a single EFS Access Point (RWO — ReadWriteOnce, as OC has only 1 replica).

HA Managed Controllers

Each managed controller (citest-1, citest2) runs as a StatefulSet with 2 replicas in active-active HA mode. Both replicas share the same EFS Access Point (ReadWriteMany). They coordinate via the shared filesystem — one replica is the primary, the other is hot standby. A PodDisruptionBudget (minAvailable=1) prevents Karpenter or cluster upgrades from evicting both replicas simultaneously. Pod anti-affinity spreads replicas across AZs, so a single AZ failure does not take down a controller.

Build Agents — Ephemeral Pod Lifecycle

Build agents are ephemeral Kubernetes pods — they exist only for the duration of a build. The controller creates the pod, Karpenter provisions an EC2 spot node if needed, the agent connects back via JNLP, the build runs, and the pod is deleted. This is fundamentally different from traditional static Jenkins agents. The agent namespace is isolated from the controller namespace — separate RBAC, ResourceQuota, and LimitRange per controller.

Agent pods declare tolerations: workload=agents:NoSchedule and nodeSelector: role=agent in the inline pod template. This ensures they land on Karpenter-provisioned agent nodes, not controller nodes. Resources: 500m CPU request / 1 CPU limit, 512Mi memory request / 1Gi limit.

Storage — Amazon EFS

All persistent Jenkins data lives on Amazon EFS — a managed, serverless, multi-AZ NFS filesystem. EFS is the only storage that supports ReadWriteMany (RWX), which is required for HA active-active controllers (both replicas must mount the same JENKINS_HOME simultaneously). One EFS Access Point is created per controller. Access Points enforce a root directory and unix UID/GID — giving each controller isolated storage within the same filesystem.

subPathPattern: how Access Points get named

The EFS CSI StorageClass uses subPathPattern: ${.PVC.name}. This means the Access Point name is deterministic and stable across pod restarts.

Secrets Management

Secrets are stored in AWS Secrets Manager and pulled into Kubernetes as native Secret objects by scripts/bootstrap.sh at cluster setup time. No secrets are stored in Git — ever. The bootstrap script uses the AWS CLI to fetch secret values, then creates idempotent Kubernetes secrets via kubectl apply.

GitHub Actions — OIDC Keyless Auth

GitHub Actions workflows authenticate to AWS using OIDC (OpenID Connect) — no long-lived AWS access keys stored in GitHub secrets. When a workflow runs, GitHub mints a short-lived JWT signed by token.actions.githubusercontent.com. AWS STS verifies the JWT against the registered OIDC provider and returns 15-minute temporary credentials. The IAM role trust policy restricts assumption to runs from Ramee09/cbci_on_aws:refs/heads/main only.
