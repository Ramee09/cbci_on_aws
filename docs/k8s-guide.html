🔐 Connecting to the Cluster

1

Verify your AWS identity

Confirm you're authenticated as naga-admin in the right account before touching the cluster.

zsh — verify identity
❯ aws sts get-caller-identity --profile cbci-lab
{
  "UserId": "AIDA...",
  "Account": "123456789012",
  "Arn": "arn:aws:iam::123456789012:user/naga-admin"
}
2

Update kubeconfig — add the cluster to kubectl

This writes the EKS cluster credentials into ~/.kube/config so kubectl can talk to it.

zsh — kubeconfig
❯ aws eks update-kubeconfig \
  --name cbci-lab \
  --region us-east-1 \
  --profile cbci-lab
Added new context arn:aws:eks:us-east-1:...:cluster/cbci-lab to /Users/naga/.kube/config
3

Verify connection to the cluster

Check that kubectl can reach the API server and see the nodes.

zsh — verify cluster
❯ kubectl cluster-info
Kubernetes control plane is running at https://XXXX.gr7.us-east-1.eks.amazonaws.com

❯ kubectl get nodes
NAME                      STATUS   ROLES   AGE   VERSION
ip-10-0-1-45.ec2.internal   Ready   <none>  2d    v1.31.0
ip-10-0-2-89.ec2.internal   Ready   <none>  2d    v1.31.0
ip-10-0-3-12.ec2.internal   Ready   <none>  1d    v1.31.0
4

Set the default namespace (optional but useful)

So you don't have to type -n cloudbees on every command.

zsh — set namespace
❯ kubectl config set-context --current --namespace=cloudbees
Context "arn:aws:eks:..." modified.

# To check current context and namespace:
❯ kubectl config get-contexts
CURRENT  NAME            CLUSTER      NAMESPACE
*        arn:aws:eks...  cbci-lab     cloudbees

■ Working with Pods

Pods are the smallest unit in Kubernetes — a running container (or group of containers) on a node. The CBCI pods live in the cloudbees namespace.
🔍

List Pods

Read
All pods in cloudbees namespace
$kubectl get pods -n cloudbees
Watch pods live (refreshes automatically)
$kubectl get pods -n cloudbees -w
Show which node each pod is running on
$kubectl get pods -n cloudbees -o wide
All pods across all namespaces
$kubectl get pods -A
📄

Describe a Pod

Detail
Full details — events, mounts, conditions
$kubectl describe pod cjoc-0 -n cloudbees
Describe a controller pod
$kubectl describe pod controller-1-0 -n cloudbees
ⓘ Describe shows Events at the bottom — the most useful section when troubleshooting a pod that won't start.
📄

Pod Logs

Logs
Live logs (tail)
$kubectl logs -f cjoc-0 -n cloudbees
Last 100 lines
$kubectl logs cjoc-0 -n cloudbees --tail=100
Specific container in a pod (e.g. casc-client)
$kubectl logs cjoc-0 -c casc-client -n cloudbees -f
Previous (crashed) container logs
$kubectl logs cjoc-0 -n cloudbees --previous
🖥

Shell into a Pod

Exec
Open a bash shell inside the OC pod
$kubectl exec -it cjoc-0 -n cloudbees -- bash
Run a single command (no shell)
$kubectl exec cjoc-0 -n cloudbees -- ls /var/jenkins_home
Shell into specific container in pod
$kubectl exec -it cjoc-0 -c casc-client -n cloudbees -- sh
⚠ Exit with exit or Ctrl+D

What a healthy pod list looks like

kubectl get pods -n cloudbees -o wide
NAME                   READY   STATUS     RESTARTS   AGE   NODE
cjoc-0                 2/2     Running     0          2d   ip-10-0-1-45.ec2.internal
controller-1-0         2/2     Running     0          2d   ip-10-0-2-89.ec2.internal
controller-1-1         2/2     Running     0          2d   ip-10-0-3-12.ec2.internal
controller-2-0         2/2     Running     0          1d   ip-10-0-2-89.ec2.internal
controller-2-1         2/2     Running     0          1d   ip-10-0-3-12.ec2.internal

# 2/2 means 2 containers running in the pod (jenkins + casc-client sidecar)
# Both replicas of each controller on DIFFERENT nodes — that's HA working correctly

💾 Storage — PVCs & EFS

Each CBCI pod mounts its own EFS Access Point as a PersistentVolumeClaim (PVC). The PVC → PV → EFS Access Point chain is what gives Jenkins pods durable storage.

💾

PVCs & PVs

Storage
List persistent volume claims
$kubectl get pvc -n cloudbees
List persistent volumes (cluster-wide)
$kubectl get pv
Describe a PVC (shows EFS access point)
$kubectl describe pvc jenkins-home-cjoc-0 -n cloudbees
🔌

StorageClass

Config
See available storage classes
$kubectl get storageclass
Describe EFS storage class
$kubectl describe storageclass efs-sc
kubectl get pvc -n cloudbees
❯ kubectl get pvc -n cloudbees
NAME                        STATUS  VOLUME        CAPACITY  ACCESS MODES  STORAGECLASS
jenkins-home-cjoc-0         Bound   pvc-abc...    20Gi      RWO          efs-sc
jenkins-home-controller-1-0 Bound   pvc-def...    20Gi      RWX          efs-sc
jenkins-home-controller-1-1 Bound   pvc-def...    20Gi      RWX          efs-sc
jenkins-home-controller-2-0 Bound   pvc-ghi...    20Gi      RWX          efs-sc
jenkins-home-controller-2-1 Bound   pvc-ghi...    20Gi      RWX          efs-sc

# RWO = ReadWriteOnce (OC — single pod)
# RWX = ReadWriteMany (controllers — both replicas share same EFS Access Point)

🔍 Logs, Events & Debugging

⚠

Events

Debug
All events in cloudbees namespace (sorted)
$kubectl get events -n cloudbees --sort-by='.lastTimestamp'
Watch events live
$kubectl get events -n cloudbees -w
ⓘ Events are the first place to look when a pod is stuck in Pending or CrashLoopBackOff.
📋

StatefulSets

Workloads
List all StatefulSets
$kubectl get statefulsets -n cloudbees
Rollout status
$kubectl rollout status statefulset/cjoc -n cloudbees
Describe a StatefulSet
$kubectl describe statefulset controller-1 -n cloudbees
🔄

ConfigMaps & Secrets

Config
List ConfigMaps
$kubectl get configmaps -n cloudbees
List Secrets (names only — never print values)
$kubectl get secrets -n cloudbees
View everything in namespace
$kubectl get all -n cloudbees

Debugging a pod stuck in Pending or CrashLoopBackOff

debug flow
# Step 1 — see the pod status
❯ kubectl get pods -n cloudbees
cjoc-0      0/2   CrashLoopBackOff  5   3m

# Step 2 — check events to see WHY
❯ kubectl describe pod cjoc-0 -n cloudbees
Events:
  Warning  BackOff  OOMKilled — container exceeded memory limit

# Step 3 — check logs of the crashed container
❯ kubectl logs cjoc-0 -n cloudbees --previous

# Step 4 — for Pending pods, check if Karpenter is provisioning a node
❯ kubectl get events -n cloudbees --sort-by='.lastTimestamp' | tail -20

⚙ CBCI-Specific Commands

Commands specific to checking the health of Operations Center, controllers, and CasC bundle delivery.

📋

Check OC & Controller Status

CBCI
All CBCI pods at a glance
$kubectl get pods -n cloudbees
OC logs (Jenkins startup)
$kubectl logs cjoc-0 -n cloudbees -f
Controller-1 logs
$kubectl logs controller-1-0 -n cloudbees -f
📦

CasC Bundle Delivery

CasC
SCM Retriever logs (OC bundle polling)
$kubectl logs cjoc-0 -c scm-retriever -n cloudbees -f
casc-client logs (controller bundle pull)
$kubectl logs controller-1-0 -c casc-client -n cloudbees -f
Check bundles landed on EFS
$kubectl exec cjoc-0 -n cloudbees -- ls /var/jenkins_home/jcasc-bundles-store/
⚡

Agent Pods

Agents
Watch agents spin up during a build
$kubectl get pods -n cloudbees -w | grep agent
Check agent node taint
$kubectl describe node <agent-node> | grep Taint
View agent pod logs
$kubectl logs <agent-pod-name> -n cloudbees
🔌

Services & Ingress

Network
List all services
$kubectl get svc -n cloudbees
List ingress (shows ALB URL)
$kubectl get ingress -n cloudbees
Describe ingress (annotations, rules)
$kubectl describe ingress -n cloudbees
