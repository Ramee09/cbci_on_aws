Personal learning lab: CBCI Modern on Amazon EKS, mirroring Mastercard production on Azure AKS

Owner: Naga Ramesh Swarna Region: us-east-1 Cluster: cbci-lab (EKS 1.31) Domain: myhomettbros.com Document: HLS v2.0 Date: April 2026

1. Executive Summary

This project builds a fully production-grade CloudBees CI (CBCI) Modern platform on Amazon EKS. It is a personal learning lab designed to mirror the architecture running at Mastercard on Azure AKS — porting every component to AWS to deeply understand both cloud platforms and Kubernetes operations.

All Jenkins/CBCI configuration is managed via Configuration as Code (CasC) bundles — no manual UI interaction. Infrastructure is Terraform. Workloads are Helm-deployed. Bundle changes flow from a Git push and are picked up automatically via the SCM Retriever polling GitHub every 3 minutes.

Key Learning Outcome The owner can explain every component — from VPC CIDR design to CBCI bundle inheritance, from Karpenter NodePool tainting to SAML SSO RBAC — well enough to field it in a senior platform engineering interview.

6. CasC Bundle Hierarchy & Git-to-Controller Flow

Engineer edits CasC YAML and pushes to GitHub (main) Files under casc/oc-bundle/ (OC config) or casc/controller-bundles/ (controller config).
GitHub Actions runs YAML validation (casc-deploy.yaml workflow) Validates every *.yaml under casc/ with yaml.safe_load. No AWS access needed — repo is public.
SCM Retriever sidecar (OC pod) polls GitHub every 3 minutes Detects new commit hash for casc/oc-bundle/ and downloads new files into /var/jenkins_home/core-casc-bundle/.
OC JCasC hot-reloads from updated bundle files bundleUpdateTiming.automaticReload=true — JCasC config applied without restart. Plugin changes trigger safe-restart.
bundleStorageService polls GitHub every 120 seconds for controller bundles Configured in OC jenkins.yaml. Reads casc/controller-bundles/ tree, writes updated bundles to $JENKINS_HOME/jcasc-bundles-store/ on EFS.
casc-client on each controller polls OC, detects version change, applies bundle Controller inherits from base-bundle (shared Kubernetes cloud config, auth) and overrides only its own systemMessage and location.url.

7. Node Architecture — Karpenter NodePools

Three compute tiers run in the same EKS cluster. Karpenter manages scale-up/scale-in for controller and agent tiers; the system tier is a fixed managed node group.

NodePoolInstance TypePricingTaint / LabelPods Scheduled
System (managed) t3.large ×2 On-demand, always-on role=system CoreDNS, kube-proxy, LBC, ExternalDNS, Karpenter, Velero
Controllers (Karpenter) t3.large / t3.xlarge On-demand (no eviction risk) role=controller OC StatefulSet, devflow StatefulSet ×2, test1 StatefulSet ×2
Agents (Karpenter) Mixed: t3.medium, t3.large, t3.xlarge Spot-first, on-demand fallback role=agent + NoSchedule taint Ephemeral Jenkins agent pods only (tolerate the taint)
Why the taint on agents? The NoSchedule taint on agent nodes ensures a rogue build job that exhausts memory/CPU cannot evict controller pods. Controllers and agents live on physically separate EC2 instances.

9. Entra ID SAML SSO Flow

🔐 Identity Provider (Entra ID)

  • Enterprise Application: CBCI-Lab
  • Protocol: SAML 2.0
  • Entity ID: https://cjoc.myhomettbros.com/securityRealm/saml2/metadata
  • ACS URL: /securityRealm/saml2/acs
  • Groups attribute: http://schemas.microsoft.com/ws/2008/06/identity/claims/groups
  • Groups: cbci-admins, cbci-developers

🔒 Service Provider (OC)

  • Plugin: Jenkins SAML Plugin (not Azure AD plugin)
  • Group-to-role mapping in OC CasC jenkins.yaml
  • cbci-admins → Admin role
  • cbci-developers → Developer role
  • Active-active HA compatible (session via EFS-backed JENKINS_HOME)
  • Local admin retained as break-glass fallback
User visits https://cjoc.myhomettbros.com OC detects unauthenticated session, generates SAML AuthnRequest.
Browser redirected to Entra ID login page User authenticates with their personal Entra tenant (MFA if configured).
Entra ID returns signed SAML Assertion to OC ACS endpoint Assertion includes group membership claims in the configured attribute.
OC maps groups → Jenkins roles, establishes session User is granted appropriate permissions. Session is EFS-backed, survives pod restarts.

10. Secret Management

Simplified to a single secret. No External Secrets Operator. No static secrets in Git.

SecretStored InDelivered ViaConsumer
jenkins-admin-password AWS Secrets Manager
cbci-lab/jenkins-admin-password		 bootstrap.sh pulls with AWS CLI,
creates K8s secret jenkins-admin-secret		 OC pod — admin login
Why no ESO? ESO adds a controller, CRDs, IRSA role, and ClusterSecretStore — significant overhead for one secret. The bootstrap.sh one-liner is equivalent operationally and requires no running controller. The trade-off is no auto-rotation push; rotation requires re-running bootstrap, which is acceptable for a lab.

12. Terraform Module Breakdown

💻 10-vpc

  • VPC 10.0.0.0/16, 3 AZs
  • 3 public subnets (ALB) + 3 private (nodes/EFS)
  • Single NAT Gateway in us-east-1a
  • VPC Endpoints: EC2, ECR, S3, STS
  • Route tables: public → IGW, private → NAT

⚙ 20-eks

  • EKS 1.31 control plane
  • Managed node group: 2× t3.large (system)
  • OIDC provider for IRSA
  • aws-auth ConfigMap for naga-admin
  • Add-ons: CoreDNS, kube-proxy, VPC CNI, EBS CSI

💾 30-efs

  • EFS filesystem (KMS-encrypted, Elastic throughput)
  • Mount targets in all 3 private subnets
  • Access Point: /oc (uid 1000)
  • Access Point: /devflow (uid 1000)
  • Access Point: /test1 (uid 1000)
  • Security group: NFS 2049 from node SG

🏭 60-platform

  • S3 buckets: Velero backup, TF state
  • AWS Secrets Manager: jenkins-admin-password
  • IAM roles: LBC, ExternalDNS, Velero (IRSA)
  • Route 53 Hosted Zone: myhomettbros.com
  • ACM Certificate: *.myhomettbros.com (DNS-validated)
  • AWS Backup vault + plan (daily, 14-day retention)
