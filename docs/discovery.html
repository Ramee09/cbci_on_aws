1
Is there an existing EKS cluster CBCI must share, or will a dedicated cluster be provisioned? If shared: what add-ons are already installed, who owns the aws-auth ConfigMap, and what namespace naming convention is enforced?
CriticalOpen

Why this matters

This is the single biggest architectural fork in the whole project. A shared cluster means CBCI is a tenant, not an owner. That has cascading effects:

  • Add-on conflicts: If AWS Load Balancer Controller (LBC) is already installed at a different version, we cannot install our own. The existing LBC's IngressClass name and webhook configuration must be used as-is, or its owner must upgrade it.
  • Karpenter: Karpenter is a cluster-scoped controller. Installing it requires cluster-admin and it manages EC2 fleet on behalf of the whole cluster — a shared-cluster team will almost certainly not permit a tenant to install it. This forces CBCI agents onto Cluster Autoscaler-backed managed node groups instead.
  • Admission webhooks: CBCI deploys mutating/validating webhooks (e.g., for pod injection). If another team already has a webhook with a conflicting name or failurePolicy: Fail, deploying CBCI can break unrelated workloads on the cluster.
  • EFS CSI Driver: If not installed, we need cluster-admin to install it. If it is installed but at an older version that doesn't support provisioningMode: efs-ap (Access Points), our per-controller storage isolation model breaks.

Architectural impact

  • Dedicated cluster: Proceed with current architecture — Karpenter NodePools for controllers and agents, full add-on control, own IngressClass. Terraform manages the full cluster lifecycle.
  • Shared cluster, Karpenter available: Coordinate with cluster owner on NodePool naming, resource limits, and EC2 node tag namespacing to avoid conflicts with other tenants.
  • Shared cluster, no Karpenter: Replace Karpenter agent pool with a Cluster Autoscaler-managed MNG. Agent scale-out is slower (2–3 min vs 30 sec). Lose bin-packing efficiency. Must negotiate MNG sizing with cluster owner.
  • Shared cluster, LBC pre-installed: Remove LBC Terraform module. Reference existing IngressClass in all Ingress manifests. Confirm existing LBC version supports all required annotations (e.g., alb.ingress.kubernetes.io/group.name).
Key things to document from this meeting
Cluster name · OIDC provider ARN · EKS version · List of installed add-ons and their versions · Who owns aws-auth / EKS access entries · Existing IngressClass names · NodePool/MNG naming convention · Maximum namespace count or resource quota per tenant

What AWS account structure is in use — single account, AWS Organizations with SCPs, or an enterprise Landing Zone (Control Tower)? Are there IAM permission boundaries applied to all roles in the account, and who can grant exceptions?
CriticalOpen

Why this matters

SCPs and permission boundaries are invisible to the caller — a denied IAM action returns the same error as a misconfigured policy, making debugging extremely difficult. CBCI requires the following IAM actions across multiple services; any one of them being SCP-denied will silently break a component:

  • IRSA / STS: sts:AssumeRoleWithWebIdentity — if denied, every pod-level AWS call fails
  • EFS CSI: elasticfilesystem:CreateAccessPoint, elasticfilesystem:DescribeMountTargets — if denied, PVC provisioning fails silently
  • Velero / S3: s3:PutObject, s3:GetObject, s3:DeleteObject — if denied, backups appear to succeed but nothing is written
  • AWS Backup: backup:CreateBackupVault, backup:StartBackupJob — if denied, EFS snapshots never run
  • Secrets Manager: secretsmanager:GetSecretValue — if denied, bootstrap script fails at the credentials-pull step
  • EC2 (Karpenter): ec2:RunInstances, ec2:DescribeInstanceTypes — if denied, build agents never scale out

Landing Zone environments often restrict cross-region operations and resource creation outside approved modules. If Terraform state is stored in S3 with a locking DynamoDB table, those must exist before any Terraform can run — and the Landing Zone may require them to be provisioned by their pipeline, not ours.

Architectural impact

  • Landing Zone with VPC factory: Do not provision VPC in Terraform. Reference existing subnet IDs and route table IDs as data sources. Confirm subnets are tagged kubernetes.io/role/internal-elb: 1 and kubernetes.io/cluster/<name>: shared — required for ALB auto-discovery.
  • SCPs present: Request a full SCP document from the cloud team before writing a single IAM policy. Map every CBCI-required action against the SCP denies. Seek explicit exceptions in writing before the sprint begins.
  • Permission boundaries on all roles: Every IAM role created by Terraform (Velero, AWS Backup, EFS CSI, Karpenter) must include a permissions_boundary ARN in the Terraform resource block. Roles without it will be rejected by the boundary policy.
  • Centrally managed Terraform state: The 55-backup and other modules may need to write state to a platform-team-managed S3 bucket rather than one we create. Update the backend "s3" block accordingly.
Key things to document from this meeting
Account ID · Organization ID if applicable · SCP document or link · Permission boundary ARN if enforced · Terraform state bucket name and key prefix · Who approves IAM exceptions · Landing Zone team contact for VPC/subnet provisioning

Is IRSA (IAM Roles for Service Accounts via OIDC) enabled on the cluster? Alternatively, has the team evaluated EKS Pod Identity (the newer replacement)? Who manages the OIDC provider registration in IAM, and is there a process to create IRSA roles?
CriticalOpen

Why this matters

CBCI requires AWS API access from three distinct workloads, each needing different permissions. Without a workload-identity mechanism (IRSA or Pod Identity), the only alternatives are:

  • Node IAM role: Attach all permissions to the EC2 worker node role — every pod on every node gets every permission. This violates least-privilege and is rejected by most enterprise security teams.
  • Static access keys in Secrets Manager: Keys that expire or are rotated require a deployment to update. A leaked key has blast radius across all uses. This is the fallback of last resort.

The three workloads that need AWS access and their required permissions:

  • Velero (velero/velero ServiceAccount) → S3 CRUD + EC2 snapshot actions on the Velero bucket only
  • EFS CSI Driver (kube-system/efs-csi-controller-sa) → elasticfilesystem:* on the CBCI EFS filesystem ARN only
  • Karpenter (kube-system/karpenter) → EC2 fleet management actions scoped to cluster-tagged resources

If the OIDC provider is registered in IAM but the team has not yet created IRSA roles for these service accounts, we need to know the approval process and lead time — IRSA role creation is the prerequisite for deploying any of the above components.

Architectural impact

  • IRSA available, self-service role creation: Terraform creates the three IRSA roles (cbci-velero, cbci-efs-csi, cbci-karpenter) with scoped trust policies. No external dependency — proceed as designed.
  • IRSA available, ticket-gated role creation: Raise IAM role creation tickets for all three roles in parallel. Provide exact trust policy JSON and permission policy JSON per role. Expect 5–10 day lead time. This is the critical path for initial deployment.
  • EKS Pod Identity available (EKS ≥ 1.29): Simpler than IRSA — no OIDC trust condition in the role. Replace IRSA annotations in Helm values with Pod Identity association resources. Terraform module changes required but architecture is improved.
  • Neither available: Document the node IAM role ARN and scope permissions to the minimum set. Accept the security trade-off formally in writing. Escalate to security team as a risk item.
Key things to document from this meeting
OIDC provider ARN (format: arn:aws:iam::ACCOUNT:oidc-provider/oidc.eks.REGION.amazonaws.com/id/HASH) · Whether Pod Identity is available · IAM role creation process and SLA · Who is the IAM approver · Any naming convention for IAM roles

What EKS version is currently running, what is the planned upgrade cadence, and does the cluster use EKS-managed add-ons (with version pinning in the console) or self-managed add-ons deployed by the platform team via Helm?
HighOpen

Why this matters

Each CBCI component has a minimum EKS version dependency. Running below those minimums causes silent failures or missing features:

  • AWS Load Balancer Controller v2.7+ requires EKS ≥ 1.25 for full IngressGroup and target group binding support
  • EFS CSI Driver v2.x with Access Point provisioning requires Kubernetes ≥ 1.23 and the CSIMigrationEFS feature gate
  • Karpenter v1.x requires EKS ≥ 1.25 and the NodePool/EC2NodeClass CRD API (v1beta1 minimum)
  • Pod Security Admission (enforced Restricted/Baseline mode) was GA in 1.25 — older clusters use PodSecurityPolicy which is removed in 1.25+

The add-on management model matters separately: EKS-managed add-ons (CoreDNS, kube-proxy, VPC CNI) are upgraded by AWS on their schedule during cluster upgrades and can conflict with self-managed versions. If the cluster team manages LBC or EFS CSI as an EKS-managed add-on, we cannot change its version without their involvement.

Architectural impact

  • EKS ≥ 1.28, self-managed add-ons: Full control — proceed with pinned Helm chart versions for LBC, EFS CSI, Karpenter as designed.
  • EKS-managed LBC or EFS CSI: Remove those components from our Terraform. Reference the managed versions. Accept that version upgrades require coordination with the cluster team — plan a compatibility check before every CBCI version upgrade.
  • Upgrade cadence every 3 months: Build an upgrade compatibility test suite. Pin all add-on versions to a known-good matrix and test on a non-prod cluster one sprint before each EKS upgrade.
  • EKS ≤ 1.24 (EOL): Escalate immediately. Running end-of-life EKS is a security risk and blocks several required features. Push for an upgrade before CBCI deployment begins.
Key things to document from this meeting
EKS version · Add-on management model (managed vs self-managed) for CoreDNS, VPC CNI, LBC, EFS CSI · Upgrade schedule (next planned upgrade date) · Who coordinates add-on version upgrades

How do engineers get kubeconfig access — is it via aws eks update-kubeconfig with SSO, a bastion/jump host, or a centralised access platform like Teleport or Boundary? And separately, how do non-human CI pipeline agents authenticate to the Kubernetes API at runtime?
HighOpen

Why this matters

There are two distinct access patterns that must both work:

  • Human access (platform engineers): Running bootstrap scripts, applying Terraform, executing velero restore, reading pod logs during incidents. If this requires interactive SSO login with a 15-minute token, any script that runs longer than 15 minutes will fail mid-execution.
  • Non-human access (CBCI pipeline agents): Jenkinsfiles that launch Kubernetes pods, check PVC status, or deploy to the cluster via kubectl inside a build. These run unattended and cannot do an interactive SSO login. They need a ServiceAccount token bound to an RBAC Role with appropriate permissions.

The Kubernetes agent plugin in CBCI uses the in-cluster ServiceAccount token by default (the token mounted at /var/run/secrets/kubernetes.io/serviceaccount/token). This requires a ClusterRole binding that allows launching pods in the agent namespace. If RBAC is managed externally (e.g., via OPA or a GitOps pipeline owned by the cluster team), adding this binding may require a ticket.

Architectural impact

  • SSO via aws eks update-kubeconfig: Token refresh every 15 minutes. Bootstrap scripts must be designed to complete within one token cycle, or the AWS CLI must be configured with SSO profile auto-refresh. Velero restore operations that take >15 min need a separate ServiceAccount token path.
  • Teleport / Boundary: Engineers access the cluster through a proxy. Running kubectl exec into CBCI pods for troubleshooting works, but the proxy session timeout must be longer than a Jenkins startup time (~3 min). Confirm proxy session TTL.
  • RBAC self-service: CBCI deploys three ServiceAccounts that need RoleBindings: jenkins (in ci-controllers, needs pod/exec/logs in ci-agents-* namespaces), velero (in velero namespace, needs cluster-scoped backup CRDs), karpenter (needs EC2NodeClass and NodePool CRD access). If RBAC is managed by the cluster team, provide the exact Role/ClusterRole YAML for them to apply.
  • Break-glass access: Document the procedure for accessing the cluster when SSO is unavailable (e.g., during an incident where Entra ID is down). This is typically a pre-generated kubeconfig with a long-lived ServiceAccount token stored in a secure vault.
Key things to document from this meeting
Access method (SSO / bastion / Teleport) · Token TTL · Who controls RBAC in the cluster · Process for adding ClusterRoleBindings · Break-glass access procedure · ServiceAccount token validity period (default 24h in EKS, can be customised)

What EC2 instance families are approved for EKS worker nodes? Is Spot instance usage permitted for ephemeral build agents? Are there any maximum vCPU or memory limits enforced per node, per namespace, or per account via Service Quotas?
HighOpen

Why this matters

Build agent capacity directly determines how many concurrent pipelines can run. Three separate constraints can limit this — and all three need to be understood before sizing the Karpenter NodePool:

  • Approved instance families: Enterprise security teams sometimes restrict certain families (e.g., GPU instances, bare-metal, or very large instances) via SCP or AWS Config rules. Karpenter will attempt to launch whatever the scheduler requests — if the instance type is SCP-denied, the node launch fails silently and the build agent pod stays pending.
  • Spot usage: If Spot is prohibited (common in regulated environments), the agent NodePool must use capacity-type: on-demand only. This affects cost significantly and means builds queue during instance capacity crunches rather than automatically shifting instance types.
  • Service Quotas: AWS accounts have default limits on vCPU count per region per instance family (e.g., 32 vCPUs for On-Demand Standard). A pipeline burst that tries to launch 20 concurrent m5.2xlarge agents (80 vCPUs) will be throttled by EC2 RunInstances and agents will hang pending. Quota increases require an AWS support case with 1–3 business day SLA.

Architectural impact

  • Approved families known: Populate Karpenter NodePool spec.requirements with explicit kubernetes.io/instance-type or karpenter.k8s.aws/instance-family allowlist values.
  • Spot blocked: Set karpenter.sh/capacity-type: on-demand in NodePool requirements. Define explicit limits.cpu and limits.memory on the NodePool to cap cluster size and avoid runaway scaling.
  • Service Quota limits: Calculate maximum concurrent build capacity against the quota. If insufficient, raise AWS Support quota increase requests before go-live. Build a Prometheus alert on karpenter_nodeclaims_terminating to detect quota-throttled scale-outs.
  • ResourceQuota on agent namespaces: Regardless of Karpenter limits, apply K8s ResourceQuota to ci-agents-controller-1 and ci-agents-controller-2 namespaces to prevent a single runaway pipeline from consuming the entire agent pool.
Key things to document from this meeting
Approved instance families (allowlist) · Spot permitted yes/no · Current EC2 vCPU quota for On-Demand Standard in us-east-1 · ResourceQuota policy per namespace · Any SCP that restricts ec2:RunInstances by instance type or size

What resource tagging standard applies to all AWS resources — what are the mandatory tag keys, allowed values for each, and is compliance enforced by an SCP deny, an AWS Config rule with auto-remediation, or a post-creation audit? Does Karpenter's tag propagation to EC2 instances need to match the same standard?
MediumOpen

Why this matters

Tags are not cosmetic — they are the control plane for cost allocation, compliance reporting, and in some SCP configurations, the gating mechanism for resource creation. Specifically:

  • SCP tag enforcement: Some enterprise SCPs include a condition like "StringNotEquals": {"aws:RequestedRegion": "us-east-1"} or require a CostCenter tag on every CreateBucket / RunInstances call. A missing or wrong-valued tag causes the API call to return AccessDenied — with no indication in the error that it's a tag issue.
  • Karpenter EC2 tags: Karpenter propagates tags from the EC2NodeClass spec to every EC2 instance it launches. If the tagging standard includes a DataClassification tag with specific values, those must be baked into the EC2NodeClass definition — Karpenter cannot dynamically set them per-pod.
  • EFS Access Points: Each CBCI controller creates an EFS Access Point (provisioned by the EFS CSI driver, not Terraform). The CSI driver does not propagate tags from the StorageClass to the Access Point. A separate tagging Lambda or Config remediation may be required.
  • S3 bucket: The Velero backup bucket created by Terraform must carry all mandatory tags at creation time — S3 bucket creation is logged by CloudTrail and audited by Config rules within minutes.

Architectural impact

  • Known mandatory tags: Add them to every Terraform resource block's tags map and to the Karpenter EC2NodeClass tags field. Create a Terraform local common_tags map and reference it everywhere.
  • Config auto-remediation: Tags added after resource creation by Config remediation may overwrite manually set values. Test tagging behaviour in a sandbox account before production deployment.
  • EFS CSI Access Point tagging gap: File a request with the storage team to either: (a) add a post-creation tagging Lambda triggered by EFS CloudTrail events, or (b) accept that Access Points will fail the Config rule and be documented as a known exception.

Common mandatory tag keys at financial institutions — confirm each

  • Application — e.g., cbci-platform
  • Environment — e.g., prod / nonprod
  • Owner — team name or email DL
  • CostCenter — finance charge-back code
  • DataClassification — e.g., Internal / Confidential
  • Compliance — e.g., SOX / PCI / None
  • SupportedBy — on-call team name

Is the VPC pre-provisioned by a Landing Zone or Network team, or can Platform Engineering provision its own? If pre-provisioned: what are the VPC ID and subnet IDs per AZ, are the subnets tagged correctly for EKS ALB discovery, and are VPC endpoints available for S3, STS, ECR, and Secrets Manager?
CriticalOpen

Why this matters

EKS and the AWS Load Balancer Controller auto-discover subnets using specific tags. Missing tags cause the ALB to not be created, or to be created in the wrong subnets (e.g., public instead of private). There is no error message — the Ingress just sits with no ADDRESS for 10+ minutes before timing out.

The required subnet tags that must exist before EKS installation:

  • Private subnets (where EKS nodes and ALB targets live): kubernetes.io/role/internal-elb: 1 and kubernetes.io/cluster/<cluster-name>: shared
  • Public subnets (where internet-facing ALBs are placed, if applicable): kubernetes.io/role/elb: 1

VPC endpoints are critical in environments where outbound internet is restricted. Without an S3 VPC endpoint, every Terraform state read and Velero backup traverses a NAT gateway — adding latency and cost. Without an STS endpoint, IRSA token exchanges go to the public internet. Without a Secrets Manager endpoint, the bootstrap script cannot pull credentials. Each endpoint requires its own Security Group rule allowing port 443 from EKS nodes.

CIDR sizing matters more than it seems: EKS assigns one IP per pod (VPC CNI). A /24 subnet (254 IPs) supports roughly 50–60 pods after system pods consume the first ~20. A cluster with 10 controller pods + 100 agent pods simultaneously needs subnets with at least 150 free IPs each, spread across 3 AZs.

Architectural impact

  • Pre-provisioned VPC with correct tags: Remove VPC Terraform module entirely. Replace with data "aws_subnet_ids" lookups using tag filters. Confirm CIDR is large enough for peak pod count.
  • Pre-provisioned VPC, missing EKS subnet tags: Request the network team to add the tags — this is a non-destructive change but requires their change process. Do not proceed with EKS installation until tags are confirmed.
  • VPC endpoints available (S3, STS, ECR, Secrets Manager): All AWS API calls from EKS pods route privately. Remove any NAT dependency for AWS service access. Update Security Groups to allow 443 from EKS node SGs to endpoint SGs.
  • No VPC endpoints, outbound internet only via NAT: Ensure NAT gateway is in each AZ or accept single-AZ NAT as a reliability trade-off. Add all AWS service endpoints to the firewall allowlist.
  • Own VPC allowed: Terraform provisions 3-AZ layout: 3 private node subnets (/22 each = ~1000 IPs), 3 public subnets (/24 each for ALBs), plus VPC endpoints for all AWS services. Validate CIDR against existing VPC peering routes.
Key things to document from this meeting
VPC ID · Subnet IDs per AZ (private and public) · Current subnet CIDR sizes and free IP count · Existing subnet tags · VPC endpoint list (S3, STS, ECR, EFS, Secrets Manager, EKS) · CIDR allocation process for new VPCs · Transit Gateway / Direct Connect attachment if present

Is outbound internet access from EKS nodes direct, via NAT, or routed through a corporate HTTP/HTTPS proxy or SSL-inspection firewall? If a proxy is in use: what is the proxy URL, does it perform TLS interception with a corporate CA certificate, and what domains are in the bypass list by default?
CriticalOpen

Why this matters

This is the most common cause of unexplained failures during CBCI deployment in enterprise environments. The failure modes are non-obvious:

  • Explicit proxy, no proxy env vars on pods: Pods ignore the proxy entirely and try to reach the internet directly. All outbound connections fail with connection refused. Jenkins plugin downloads fail silently — the plugin manager just shows "failed" with no proxy-related error.
  • TLS inspection (MITM proxy): The proxy presents its own certificate for every HTTPS connection. The JVM inside the Jenkins container only trusts its built-in CA bundle. Any connection to GitHub, updates.jenkins.io, or AWS APIs fails with SSLHandshakeException: PKIX path validation failed. Fix requires importing the corporate CA cert into the JVM truststore inside the container image — this means building a custom CBCI image or using an init container.
  • Proxy bypass list missing AWS endpoints: If STS, S3, or Secrets Manager endpoints are not in the proxy bypass list (or not in NO_PROXY), IRSA token exchanges traverse the proxy. This can cause 502 errors from the proxy as it cannot cache STS JSON responses correctly.
  • Node bootstrap via proxy: Karpenter launches new EC2 nodes that must bootstrap themselves (pull kubelet, CNI plugins, etc.) before joining the cluster. If the bootstrap user-data does not include the proxy config, the new node never joins and build agents remain pending indefinitely.

Architectural impact

  • Explicit proxy required: Configure HTTP_PROXY, HTTPS_PROXY, and NO_PROXY in: (1) Karpenter EC2NodeClass userData block for node bootstrap, (2) CBCI OC and controller pod env vars in Helm values, (3) Jenkins JVM args via JAVA_OPTS=-Dhttps.proxyHost=... -Dhttps.proxyPort=..., (4) casc-retriever container env vars for GitHub API calls.
  • TLS inspection CA cert: Build a custom CBCI container image that adds the corporate CA to the JVM truststore (keytool -importcert in the Dockerfile). Or use an init container that copies the CA to a shared volume and a JVM startup script reads it. This is the most operationally fragile solution — every CBCI version upgrade requires rebuilding the custom image.
  • NO_PROXY must include: localhost,127.0.0.1,10.0.0.0/8,.svc,.svc.cluster.local,.internal,169.254.169.254,sts.amazonaws.com,s3.amazonaws.com,secretsmanager.us-east-1.amazonaws.com,elasticfilesystem.us-east-1.amazonaws.com
  • Air-gapped (no internet at all): Mirror the CloudBees plugin update center to an internal HTTP server. Mirror all container images to internal ECR. Configure Jenkins to use the internal update center URL via CasC updateCenter.sites.

Full list of external domains CBCI must reach (for firewall allowlist)

  • updates.jenkins.io, www.jenkins.io — plugin update center and UC metadata
  • api.github.com, github.com, raw.githubusercontent.com — SCM, webhook receipt, casc-retriever
  • downloads.cloudbees.com — CloudBees-specific update center (if licensed UC in use)
  • *.amazonaws.com — S3, STS, Secrets Manager, EFS, ECR, EKS APIs
  • public.ecr.aws — if pulling Velero or LBC images from public ECR
  • registry.k8s.io — Kubernetes system component images (pause container, etc.)

What is the approved ingress pattern for internal enterprise applications — AWS ALB via the Load Balancer Controller, Nginx Ingress Controller, Istio Gateway, or something else? Is the load balancer internet-facing or internal-only? Who controls TLS certificate provisioning, and is AWS ACM available?
CriticalOpen

Why this matters

CBCI's SAML SSO relies on a specific HTTPS redirect flow. The redirect URL (SAML ACS URL) is hardcoded into the IdP Enterprise Application registration. If the ingress pattern changes post-go-live, the IdP registration must be updated — a ticket process with a potentially long SLA. Getting this right at design time is critical.

The ingress pattern also determines how sticky sessions are implemented. CBCI active-active HA requires that once a user's browser is directed to a specific controller replica, all subsequent requests in that session go to the same replica. ALB achieves this with application-based stickiness (AWSALB cookie). Nginx achieves it with nginx.ingress.kubernetes.io/affinity: cookie. Istio achieves it with VirtualService consistent hashing. If stickiness is not configured correctly, users get intermittent 401 errors as the load balancer round-robins between replicas that each have their own session store.

TLS matters because the SAML metadata XML exchanged with the IdP includes the ACS URL protocol. HTTP is not acceptable — IdPs reject it. ACM certificates are free and auto-renewing; an internal PKI certificate requires a renewal process and pipeline.

Architectural impact

  • AWS ALB via LBC (preferred): Current design — Ingress object with kubernetes.io/ingress.class: alb, ALB stickiness annotation, ACM cert ARN annotation. LBC creates and manages the ALB automatically from the Ingress spec. WAF association available via annotation.
  • Nginx Ingress Controller pre-installed: Replace all Ingress annotations blocks — remove ALB-specific ones, add Nginx-specific ones (nginx.ingress.kubernetes.io/affinity: cookie, nginx.ingress.kubernetes.io/proxy-read-timeout: 600 for long-running Jenkins API calls). TLS must be configured via Nginx TLS secret, not ACM.
  • Istio Gateway: Replace Ingress entirely with Gateway + VirtualService resources. SAML redirect handling needs the x-forwarded-for header preserved. Configure consistent hashing for HA sticky sessions. Significant YAML rewrite — plan 1 sprint for this change.
  • Internal-only load balancer: SAML SSO redirect URLs must be internal hostnames. IdP Conditional Access policies may block SAML from non-Mastercard-IP sources. GitHub webhooks cannot reach an internal ALB — must configure a relay or use polling only.
  • TLS via internal PKI: Import the certificate as a Kubernetes TLS Secret. Set a reminder for renewal date (typically 1 year). Build a pipeline that auto-renews and updates the secret. This is significantly more operational overhead than ACM.
Key things to document from this meeting
Approved ingress controller and version · Internet-facing or internal-only · TLS certificate source (ACM / internal PKI / cert-manager) · Certificate renewal process · Sticky session mechanism · Any WAF requirement

What CNI is installed on the cluster, and does it enforce Kubernetes NetworkPolicies? If the CNI is AWS VPC CNI: is Calico or Network Policy Controller (NPC) also installed? Are there any pre-existing cluster-wide NetworkPolicies or Cilium policies that could affect traffic between namespaces?
HighOpen

Why this matters

CBCI active-active HA controllers use Hazelcast for in-memory session replication on TCP port 5701. The two replicas of a controller must be able to reach each other on this port. The OC must be able to reach both replicas on port 8080 for health checks and bundle pushes. Build agents must be able to reach the controller on port 50000 (Jenkins agent protocol).

These are not standard web ports. Many enterprise CNI configurations start with a default-deny posture — all traffic between pods in different deployments is blocked unless explicitly allowed. If NetworkPolicies are enforced and a default-deny base policy exists in the namespace, deploying CBCI without the correct NetworkPolicy manifests will result in:

  • Hazelcast failing to form a cluster (both replicas think they are the primary — split-brain)
  • OC failing health checks on controllers (all controllers appear offline in OC UI)
  • Build agents connecting to the controller then immediately disconnecting (port 50000 blocked)

AWS VPC CNI alone does not enforce NetworkPolicies. A cluster using VPC CNI without Calico or the Amazon VPC CNI Network Policy Controller has no NetworkPolicy enforcement — all pods can talk to all other pods by default. In this case, NetworkPolicy manifests can still be applied (they are accepted by the API server) but have no effect.

Architectural impact

  • VPC CNI alone (no enforcement): NetworkPolicy manifests are cosmetic. No action needed for connectivity, but document the lack of enforcement as a security gap. Recommend the cluster team install NPC or Calico.
  • VPC CNI + Calico or NPC: Apply CBCI NetworkPolicy manifests that explicitly allow: TCP 5701 between same-controller replicas (using com.cloudbees.cje.tenant label selector), TCP 8080 from OC to controllers, TCP 50000 from agent namespace to controller namespace, TCP 443 from all pods to K8s API server.
  • Cilium: Use CiliumNetworkPolicy instead of standard NetworkPolicy. Cilium supports L7 policies — consider restricting the Hazelcast port to only accept connections from within the same controller's pod label, not just the same namespace. More secure but requires Cilium-specific YAML.
  • Istio mTLS: Hazelcast TCP gossip is not HTTP — Istio cannot inspect or route it with VirtualService. Must configure Istio in pass-through mode for port 5701. Add a PeerAuthentication policy that excludes port 5701 from mTLS enforcement to prevent Istio sidecar interception breaking the gossip protocol.

Why this matters

Why this matters

There are three separate SSO session-related behaviours that interact in non-obvious ways for CBCI active-active HA:

  • forceAuthn=true: The Jenkins SAML plugin sends this flag in every AuthnRequest by default. It tells the IdP to force the user to re-authenticate even if they have an existing SSO session. The security rationale is preventing session hijacking. However, some IdPs (notably ADFS pre-2019 and some PingFederate configurations) return HTTP 400 when they receive this attribute.   • CBCI HA and session routing: CBCI active-active HA with 2 replicas means the SAML AuthnRequest comes from replica A, the IdP redirects the browser to the ACS URL, but the ALB may route the response to replica B. Replica B has no record of the pending AuthnRequest and returns a SAML validation error. This is solved by ALB sticky sessions — but sticky sessions must be confirmed working with the specific IdP redirect pattern, which varies by IdP.
  • Session timeout alignment: CBCI has its own session timeout (configurable, default 30 min). If the IdP session timeout is shorter (e.g., Conditional Access sign-in frequency set to 1 hour), users may be redirected to IdP mid-work without warning. If the CBCI session timeout is shorter than the IdP session, users get kicked out of CBCI and back to the IdP login page frequently.

  • forceAuthn rejected by IdP: Set forceAuthn: false in CasC jenkins.yaml SAML configuration. Accept that users with an existing IdP session won't be prompted to re-authenticate when accessing CBCI. Document as a known security trade-off.

Why this matters

CBCI stores credentials in Jenkins Credential Store, which is backed by the encrypted credentials.xml on JENKINS_HOME. The source of those credentials (where the actual secret values come from) determines the CasC configuration and the runtime retrieval mechanism. Getting this wrong means either: credentials are hardcoded in CasC YAML (a git secret exposure risk) or credentials are not available at pipeline runtime (build failures).

The three viable patterns for CBCI credential injection:

  • CasC with variable interpolation: Credentials are defined in jenkins.yaml using ${ENV_VAR} syntax. The actual values come from env vars injected into the Jenkins pod at startup. Why this matters

CBCI active-active HA requires ReadWriteMany (RWX) storage — both replicas of a controller must mount the same JENKINS_HOME simultaneously and write to it concurrently. EFS is the only AWS-native RWX solution at scale. Without it, HA is architecturally impossible and controllers must run as single replicas, eliminating the availability guarantee.

EFS has three configuration dimensions that affect CBCI performance:

  • Performance mode: General Purpose supports up to 35,000 IOPS and is appropriate for CBCI (metadata-heavy Jenkins workloads). Max I/O mode has higher aggregate throughput but adds latency — do not use it for CBCI.
  • Throughput mode: Elastic (recommended) automatically scales throughput with I/O — ideal for Jenkins' spiky build patterns. Bursting mode gives baseline 50 KiB/s per GB and can burst to 100 MiB/s — a small filesystem (20 GiB) gets very little burst capacity and will throttle during builds. Provisioned gives fixed throughput regardless of size.
  • Mount target AZ coverage: If EKS nodes in AZ us-east-1c have no EFS mount target in that AZ, pods scheduled there cannot mount the PVC. The pod will hang in a ContainerCreating state with a mount timeout. Mount targets must exist in every AZ where controller or agent nodes can run.

Architectural impact

  • EFS approved, platform creates their own: Terraform provisions the filesystem with General Purpose + Elastic throughput + KMS encryption. Creates one mount target per subnet (one per AZ). Creates one Access Point per controller. Proceed with full HA design.

Verify the security group on the filesystem allows port 2049 (NFS) from the EKS node security group. Do not create a new filesystem — reference the existing one in the StorageClass fileSystemId parameter.
  • EFS not available (blocked by SCP): Evaluate FSx for NetApp ONTAP (also RWX, higher cost, different CSI driver) or accept single-replica controllers with a documented availability trade-off. Single-replica means a rolling restart causes ~3 minutes of downtime — acceptable in some orgs, not in others.

A naïve Fluent Bit configuration that parses each line as a separate event will ingest 50 separate log events instead of 1 grouped exception. In Splunk, this makes searching for exceptions extremely difficult — you can't set an alert on "exception count > 5/min" because each line is a separate event.
  • Mix of structured and unstructured lines: Jenkins mixes INFO hudson.triggers.SCMTrigger - SCM polling started on CBCI (structured) with Exception in thread "main" java.lang.NullPointerException (unstructured). The Fluent Bit parser must handle both.
  • Build log volume: Each active build produces a continuous log stream. 50 concurrent builds × average 10 lines/second = 500 log events/second. The Fluent Bit buffer and memory limits must be sized for this peak volume, or logs are dropped during build bursts.
  • Required log schema mismatch: If the SIEM requires a field named event.timestamp (ECS format) but Fluent Bit emits time, every log event will fail schema validation and be quarantined. The Fluent Bit [FILTER] Name modify plugin can rename fields — but the target schema must be known before configuration.

Architectural impact

  • Splunk with HEC: Configure Fluent Bit [INPUT] Name tail for container logs. Apply a multiline parser using docker_no_time mode with a custom multiline rule: start_state=/^\d{4}-\d{2}-\d{2}/ to group stack traces. Configure [OUTPUT] Name splunk with HEC token, host, port, and sourcetype. Set splunk_send_raw: on to preserve the original log structure. Buffer size: set storage.type: filesystem with at least 500MB buffer to handle build log bursts.
  • CloudWatch Logs: Use the AWS for Fluent Bit (aws-for-fluent-bit) image which includes the CloudWatch output plugin natively. Configure Log Group naming: /eks/cbci-lab/ci-controllers. Set Log Group retention to match compliance requirements (90 days minimum). The CloudWatch output does not support multiline grouping — each stdout line is a separate CloudWatch log event. For stack traces, use CloudWatch Insights regex queries to group by thread prefix.
