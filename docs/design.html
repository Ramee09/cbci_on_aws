Executive Summary

Platform Overview

  • Enterprise CI/CD platform on Amazon EKS (us-east-1, 3 AZs)
  • CloudBees CI (Modern) — Operations Center + 2 managed controllers
  • Both controllers in active-active HA (2 replicas each, cross-AZ)
  • GitOps-first: zero UI config — all config is CasC in Git
  • Entra ID SAML 2.0 SSO with group-based RBAC

Architecture Pillars

  • Zero-trust secrets: AWS Secrets Manager + no static creds in Git
  • Keyless CI auth: GitHub Actions OIDC — no long-lived AWS keys
  • Cloud-native scaling: Karpenter spot agents, on-demand controllers
  • Resilient storage: Amazon EFS RWX — survives node or AZ failure
  • Data protection: AWS Backup (EFS) + Velero (K8s resources)

Key Metrics

  • Config propagation: < 3 minutes (git push → live)
  • Agent provisioning: ~30 seconds (Karpenter spot EC2)
  • GHA credential lifetime: 15 minutes (OIDC temp creds)
  • Controller HA: minAvailable=1 guaranteed by PDB
  • EFS backup RPO: 24h daily · RTO: < 1h (EFS restore)

Compliance & Governance

  • All secrets KMS-encrypted at rest (EFS + Secrets Manager)
  • No plaintext credentials in Git history
  • IAM least-privilege: GHA role permits eks:DescribeCluster only
  • SAML forceAuthn: fresh assertion required per session
  • All K8s manifests version-pinned — no latest tags

End-to-End Flow — "Engineer pushes a CasC change to GitHub"
1
Platform Engineer
Pushes a commit to main — e.g. edits casc/oc-bundle/jenkins.yaml or a controller bundle. Commit lands on GitHub.
↓ GitOps LLD
2
casc-validate (GHA)
On every PR: YAML syntax check, bundle required-fields check (id / version / apiVersion), controller cross-reference, unresolved ${VAR} scan. Blocks merge if any check fails.
↓ GHA OIDC LLD
3
controller-namespace-sync (GHA)
Fires on push to main if casc/oc-bundle/items.yaml changed. GitHub mints OIDC JWT → STS returns 15-min temp creds → sync-agent-namespaces.sh creates/updates ci-agents-controller-1 and ci-agents-controller-2 namespaces with RBAC, ResourceQuota, LimitRange.
↓ GHA OIDC LLD
4
casc-retriever (OC pod)
Polls GitHub every 1 minute. On new commit: git clone (shallow) → replaces bundle.yaml version field with commit SHA (ocBundleAutomaticVersion) → writes OC bundle to /var/jenkins_config/oc-casc-bundle/.
↓ GitOps LLD
5
git-sync sidecar (OC pod)
Simultaneously, the git-sync sidecar pulls the full repo every 60 s into a shared emptyDir volume. casc/controller-bundles/ is always current — used by bundleStorageService in the next step.
↓ GitOps LLD
6
ConfigurationUpdaterHelper (OC JVM)
OC polls its bundle directory every 2 minutes. When the checksum changes it hot-reloads the OC bundle — applies jenkins.yaml (security realm, authorization, system config), plugins.yaml, items.yaml — without pod restart.
↓ OC LLD
7
bundleStorageService (OC JVM)
Polls the git-sync volume via localFolderRetriever every 120 s. On checksum change it pushes the updated controller bundle to the affected managed controller(s) via CBCI internal API.
↓ OC LLD
8
Managed Controller (HA)
Both HA replicas hot-reload the updated bundle. PodDisruptionBudget (minAvailable=1) ensures at least one replica stays live during any rolling restart or Karpenter node drain. Config change is now live.
↓ HA LLD
9
User authenticates (SAML)
User opens https://cjoc.myhomettbros.com. ALB routes to SSO Relay → SAML AuthnRequest (forceAuthn=true) to Entra ID → user authenticates → Entra returns assertion with groups → OC maps group OID to Jenkins role → session cookie + ALB sticky.
↓ Identity LLD
10
Build Agent lifecycle
User triggers a pipeline. Controller creates agent pod in ci-agents-controller-1 with toleration: workload=agents. Karpenter detects unschedulable pod → provisions spot EC2 (~30s) → agent connects via JNLP → build runs → pod deleted → node terminates if idle.
↓ Agents LLD

GitOps / CasC Bundle Delivery
Architecture Decision Record
DecisionDual-path CasC delivery: casc-retriever for OC bundle + git-sync + bundleStorageService for controller bundles. No GitHub webhooks.
Alternatives(a) Webhooks from GitHub to OC — blocked: Jenkins RootAction endpoint rejects anonymous POST. (b) Single path via casc-retriever for all bundles — doesn't support controller bundle hot-push. (c) ConfigMap-mounted bundles — no GitOps, requires kubectl apply per change.
RationalePolling is idempotent and survives network blips. ocBundleAutomaticVersion eliminates manual version bumping. The git-sync emptyDir pattern decouples GitHub availability from OC restart cycles. Max 3-min lag is acceptable for platform config changes.

All Jenkins configuration is code in this Git repo — no UI clicking. Two parallel delivery paths exist. Path A — OC bundle via the casc-retriever Quarkus sidecar. Path B — controller bundles via the git-sync sidecar + OC's bundleStorageService. Total propagation from git push to live config: up to 3 minutes.

📄helm/values-oc.yaml:L11-L20 — CasC retriever config 📄casc/oc-bundle/bundle.yaml:L1-L13 — bundle + ocBundleAutomaticVersion 📄controller-bundles/controller-1/bundle.yaml:L4-L6 — parent inheritance 📄casc/oc-bundle/jenkins.yaml:L91-L110 — bundleStorageService config

How ocBundleAutomaticVersion works

The casc-retriever replaces the version field in bundle.yaml with the current git commit SHA before writing to disk. OC sees every push as a unique version — any push triggers reload. You never manually bump the version number.

  • To force immediate OC bundle reload: kubectl rollout restart statefulset/cjoc -n ci-controllers — casc-retriever picks up HEAD on next startup.
  • To debug CasC reload: check OC pod logs for ConfigurationUpdaterHelper and casc-retriever containers separately.

Networking & Ingress
Architecture Decision Record
DecisionSingle NAT Gateway (us-east-1a only). AWS Load Balancer Controller with ALB as ingress. ACM for TLS. All CBCI ingresses share one ALB group (cbci-oc).
Alternatives(a) NAT Gateway per AZ — full HA for outbound egress, higher operational overhead. (b) NGINX ingress — not AWS-native, requires separate NLB, adds another control plane component. (c) Self-managed TLS — manual certificate rotation burden, no auto-renewal.
RationaleConsolidated NAT is acceptable — AZ failure is an accepted risk for egress only; inbound HA is fully provided by the multi-AZ ALB. ALB group pattern consolidates OC and all controller ingresses behind one load balancer, reducing listener rule complexity. ACM auto-renews — zero operational toil.

VPC spans 3 Availability Zones. EKS nodes sit in private subnets. All inbound traffic enters through the ALB (managed by AWS Load Balancer Controller). TLS is terminated at the ALB using ACM. A single NAT Gateway in one AZ handles all outbound traffic.

📄helm/values-oc.yaml:L114-L134 — OC ALB ingress annotations 📄helm/values-oc.yaml:L137-L157 — Controller ingress annotations

Why sticky sessions on the ALB?

SAML login state is in-memory per OC pod. Without stickiness, the ALB could route the /finishLogin callback to a different pod than the one that generated the AuthnRequest → 403. Cookie-based stickiness (lb_cookie, 24h) pins each user's session to one pod.

  • ALB is provisioned by the AWS Load Balancer Controller — never create or modify it manually in the AWS console. Changes go in Helm values or Ingress annotations.

  • To add a new controller ingress: add the controller to casc/oc-bundle/items.yaml. CBCI auto-creates the Ingress using the Master.Ingress annotations from Helm values.

Identity & SSO (Entra ID SAML 2.0)
Architecture Decision Record
DecisionSAML 2.0 with Microsoft Entra ID via CloudBees SSO Relay (Quarkus). Group-based RBAC using Entra group Object IDs. HTTP-POST binding. forceAuthn=true.
Alternatives(a) OIDC with Entra — OC OIDC plugin has known session state issues in active-active HA; SAML is the CloudBees-recommended HA path. (b) Azure AD plugin — vendor-specific, less portable, deprecated path. (c) Local users only — not enterprise-grade, no centralized identity lifecycle.
RationaleSAML 2.0 is the production-proven pattern for CBCI HA. The SSO Relay decouples the SAML SP logic from OC's JVM — multiple relay replicas handle the AuthnRequest/ACS flow independently. ALB sticky sessions ensure the callback lands on the same relay pod that issued the request.

Authentication uses Microsoft Entra ID as the SAML 2.0 IdP. The SSO Relay (Quarkus, 2 replicas) handles the SAML SP flow — builds the AuthnRequest, validates the assertion, and passes the authenticated user to OC. Group membership arrives as Entra group Object IDs in the groups claim.

📄casc/oc-bundle/jenkins.yaml:L7-L16 — SAML securityRealm config 📄helm/values-oc.yaml:L159-L180 — SSO Relay deployment config

Why SAML not OIDC?

SAML is preferred for CBCI active-active HA. The SAML plugin handles cross-replica session state better. It also mirrors enterprise production patterns and is more portable across IdPs.

forceAuthn: true — why it matters

Without this, Entra can return a cached assertion. If that assertion is older than maximumAuthenticationLifetime, CBCI throws SAMLAuthnInstantException. forceAuthn: true forces Entra to issue a fresh assertion every login.

Operational Notes
  • SAML metadata refresh: if Entra rotates its signing certificate, update the IdP metadata XML in casc/oc-bundle/jenkins.yaml and push to Git — casc-retriever propagates within 1 minute.
  • SSO Relay logs: kubectl logs -n ci-controllers -l app=cjoc-ssorelay. Look for SAMLResponse validation failed to diagnose assertion issues.
  • Emergency local admin: if SSO is broken, the local admin account (password from jenkins-admin-secret) bypasses SAML. Access via direct pod port-forward: kubectl port-forward svc/cjoc 8080:80 -n ci-controllers.
  • Group RBAC: Entra group Object IDs are mapped in jenkins.yaml authorization matrix. If a user reports missing permissions, verify their Entra group membership and the OID mapping in CasC.

EKS Cluster & Karpenter Node Pools
Architecture Decision Record
DecisionHybrid strategy: fixed Managed Node Group (MNG) for system/OC pods + Karpenter for controllers (on-demand) and agents (spot with OD fallback). Three NodePools with node labels and taints for hard isolation.
Alternatives(a) All Karpenter — no fixed nodes; Karpenter churn could evict OC during consolidation. (b) All MNG — no spot savings; no elastic scaling for bursty agent workloads. (c) Separate EKS clusters for agents — extreme complexity, cross-cluster networking overhead.
RationaleMNG for system/OC provides a stable, always-on foundation without Karpenter consolidation risk. Karpenter handles elastic demand. The workload=agents:NoSchedule taint is the critical guard — without it, Karpenter spot interruptions could terminate controller nodes mid-build.

Hybrid node strategy: a fixed Managed Node Group for critical system pods, and Karpenter for dynamic controller + agent nodes. Three distinct NodePools enforce isolation via node labels and taints, preventing agent churn from evicting controller pods.

Why separate agent node pool?

Agent pods are ephemeral. Karpenter terminates idle spot nodes aggressively. Without the taint, a churning agent node could evict controller pods — breaking active builds and violating HA. The taint (workload=agents:NoSchedule) ensures only pods with the matching toleration land on agent nodes.

Operational Notes
  • Karpenter consolidation on controllers pool is set to WhenEmpty only — it will NOT evict running controller pods. This is critical: WhenEmptyOrUnderutilized must never be used for the controllers pool.
  • To drain a controller node safely: cordon it first (kubectl cordon <node>), wait for PDB to confirm safe eviction window, then drain. The PDB ensures one replica stays live.
  • Spot interruption: Karpenter handles EC2 spot interruption notices (2-min warning) automatically for agent nodes. Controller nodes use on-demand — no spot interruption risk.
  • Node pool sizing: if controller pods are Pending due to insufficient CPU/memory, review the instance type requirements in nodepool-controllers.yaml.

Operations Center — Internals
Architecture Decision Record
DecisionOC runs as 1 replica with EFS RWO. Four containers: Jenkins JVM (main), casc-retriever (Quarkus sidecar), git-sync (sidecar), cleanup-stale-config (init). numExecutors: 0 — OC is control plane only.
Alternatives(a) OC HA (2 replicas) — not supported by CloudBees; OC is a singleton control plane. (b) No casc-retriever + manual ConfigMap mounts — requires kubectl apply per CasC change, not GitOps. (c) External git-sync only (no casc-retriever) — can't deliver OC bundle with version auto-stamping.
RationaleThe dual-sidecar pattern (casc-retriever for OC bundle, git-sync for controller bundles) is the CloudBees-recommended GitOps approach for OC 2.x. The init container cleans stale XML files from older CBCI plugin versions that cause XStream CannotResolveClassException and MissingFieldException on startup.

The OC pod runs 4 containers: main Jenkins JVM, casc-retriever Quarkus sidecar, git-sync sidecar, and a one-shot cleanup init container. OC is the CBCI control plane — provisions managed controllers, distributes CasC bundles, and manages licenses. Backed by a single EFS Access Point (RWO — 1 replica).

HA Managed Controllers
Architecture Decision Record
DecisionActive-active HA: 2 replicas per controller, cross-AZ pod anti-affinity, shared EFS Access Point (RWX), PodDisruptionBudget minAvailable=1, Hazelcast TCP gossip on port 5701 for JVM cluster membership.
Alternatives(a) Active-passive (1 replica, manual failover) — no HA SLO; single pod failure = outage. (b) EBS RWO storage — no multi-attach; cannot support 2 replicas sharing JENKINS_HOME. (c) StatefulSet with 1 replica + fast restart — MTTR measured in minutes, not seconds.
RationaleEFS RWX is the enabling primitive for CBCI active-active. Both replicas write to the same JENKINS_HOME — CBCI's internal locking handles concurrent writes. Hazelcast gossip uses K8s pod discovery (DNS) to form the cluster automatically. The PDB ensures rolling Karpenter node updates never leave the controller with 0 replicas.

Each managed controller runs as a StatefulSet with 2 replicas in active-active HA mode. Both replicas share one EFS Access Point (ReadWriteMany). A PodDisruptionBudget prevents simultaneous eviction. Pod anti-affinity spreads replicas across AZs so a single AZ failure does not take down a controller.

Build Agents — Ephemeral Pod Lifecycle
Architecture Decision Record
DecisionEphemeral Kubernetes pods as build agents, isolated in per-controller namespaces (ci-agents-controller-1). Karpenter spot nodes with workload=agents:NoSchedule taint. Namespaces sync'd via GHA OIDC on items.yaml change.
Alternatives(a) Static agent VMs — no elastic scaling; idle VMs waste cost. (b) Docker-in-Docker — security risk; DinD requires privileged pods. (c) Agents in the same namespace as controllers — no RBAC isolation; a runaway build could affect controller pods.
RationaleNamespace isolation is the production security boundary. Each controller's agents are in their own namespace with their own ServiceAccount, ResourceQuota, and LimitRange. The jenkins ServiceAccount has only the permissions needed to create/delete pods — no cluster-wide access.

Agents are ephemeral Kubernetes pods — exist only for the duration of a build. Controller creates the pod, Karpenter provisions a spot EC2 node if needed, agent connects via JNLP, build runs, pod is deleted. Agent namespaces are isolated from the controller namespace — separate RBAC, ResourceQuota, LimitRange per controller.

Storage — Amazon EFS
Architecture Decision Record
DecisionAmazon EFS with one Access Point per controller. General Purpose performance mode, Elastic throughput, KMS encryption. StorageClass efs-ap with subPathPattern: ${.PVC.name}. reclaimPolicy: Retain.
Alternatives(a) EBS gp3 — RWO only; blocks HA active-active (cannot share across nodes). (b) EFS without Access Points — all controllers share root; no isolation; UID/GID collision risk. (c) Bursting throughput mode — throughput tied to storage size; Jenkins I/O is spiky and benefits from Elastic auto-scaling.
RationalesubPathPattern: ${.PVC.name} creates a deterministic Access Point path from the PVC name — stable across pod restarts and predictable for disaster recovery. Retain reclaim policy means data survives accidental PVC deletion. Elastic throughput auto-scales with Jenkins build load without pre-provisioning.

All persistent Jenkins data lives on Amazon EFS — serverless, multi-AZ NFS. EFS is the only AWS storage that supports ReadWriteMany (RWX), required for HA active-active controllers. One EFS Access Point per controller enforces isolated root directories and unix UID/GID.

StorageClass uses subPathPattern: ${.PVC.name}. When PVC controller-1-jenkins-home-controller-1-0 is created, the CSI driver creates an EFS Access Point rooted at that path. The name is deterministic — stable across pod restarts, predictable for disaster recovery.

Operational Notes
  • PVC lifecycle: because reclaimPolicy: Retain, deleting a PVC does NOT delete the underlying EFS Access Point or data. The Access Point must be manually deleted in AWS console if you want to reclaim storage.
  • EFS throughput monitoring: check CloudWatch metric StorageBytes and DataWriteIOBytes for the EFS filesystem. If throughput consistently hits Elastic limits, consider enabling Provisioned throughput mode.
  • Access Point UID/GID 1000: Jenkins runs as UID 1000 inside the container. The EFS Access Point enforces posixUser UID/GID 1000 — no permission issues on fresh PVC creation.
  • Disaster recovery: EFS Access Points can be restored from AWS Backup recovery points (see Section 11). The subPathPattern naming ensures the restored Access Point path matches the PVC name exactly.

Architecture Decision Record
DecisionAWS Secrets Manager → K8s native Secrets via bootstrap.sh. No External Secrets Operator. Secrets are pulled once at cluster setup and live in K8s. Pod env vars reference K8s Secrets via secretKeyRef.
Alternatives(a) External Secrets Operator (ESO) — syncs Secrets Manager → K8s continuously; adds complexity (CRDs, operator pods, IAM IRSA). Secrets live in AWS Secrets Manager. They are pulled into Kubernetes as native Secret objects by scripts/bootstrap.sh at cluster setup. No secrets are in Git. The bootstrap script uses the AWS CLI + kubectl apply --dry-run pattern for idempotency.

Architecture Decision Record
DecisionGitHub Actions OIDC: GitHub mints JWT → sts:AssumeRoleWithWebIdentity → 15-min temp creds. IAM role cbci-github-actions restricted to repo:Ramee09/cbci_on_aws:ref:refs/heads/main. Permission: eks:DescribeCluster only.

RationaleOIDC keyless auth is the GitHub-recommended, AWS-recommended pattern for CI/CD. No static credentials means no rotation burden and no credential exfiltration risk. The trust condition sub-claim restricts to main branch pushes only — PRs from forks cannot assume the role.

GHA workflows authenticate to AWS via OIDC — no long-lived AWS keys in GitHub secrets. GitHub mints a short-lived JWT. AWS STS verifies it against the registered OIDC provider and returns 15-minute temp credentials. The IAM role trust policy restricts assumption to runs from Ramee09/cbci_on_aws:refs/heads/main only.

Backup & Restore
Architecture Decision Record
DecisionTwo-layer backup: (1) AWS Backup for EFS filesystem data (JENKINS_HOME) — daily snapshots, 30-day retention. (2) Velero for Kubernetes resource snapshots (StatefulSets, PVCs, ConfigMaps, Secrets) — daily, stored in S3.
Alternatives(a) EFS-only backup — covers data but not K8s resource definitions; full DR requires manual recreation of all K8s objects. (b) Velero only — backs up PVC metadata but not EFS data contents; restore creates empty volumes. (c) Manual snapshots — error-prone, not scheduled, operational toil.
RationaleA DR event requires both: (1) the data (Jenkins jobs, build history, plugin configs — in EFS) and (2) the K8s objects (StatefulSets, PDBs, Ingresses — in Velero). Two layers provide independent recovery paths. AWS Backup is fully managed, integrates with KMS. Velero is the CNCF-standard K8s backup tool.

Backup covers two independent layers: Amazon EFS (Jenkins home data) via AWS Backup, and Kubernetes resource state via Velero. Both layers must be restored together for a complete disaster recovery. RPO is 24 hours (daily backup schedule). RTO target is < 1 hour for EFS restore + < 15 minutes for K8s resource restore.

Restore Runbook — Full DR

Step 1 (K8s resources): velero restore create --from-backup <backup-name> --include-namespaces ci-controllers,ci-agents-controller-1,ci-agents-controller-2
Step 2 (EFS data): In AWS Backup console → Recovery Points → select latest → Restore → select target EFS filesystem → Restore to root.
Step 3: Verify EFS Access Point paths match PVC names (subPathPattern naming).
Step 4: Scale up OC StatefulSet: kubectl scale statefulset/cjoc --replicas=1 -n ci-controllers
Step 5: Verify at /cjoc/login and check controller provisioning status in OC UI.

Operational Notes
  • Test restores quarterly: restore to a scratch namespace to verify backup integrity. An untested backup is not a backup.
  • Velero requires an S3 bucket and IAM role with S3 read/write. Use IRSA (IAM Roles for Service Accounts) — not static keys — for Velero's S3 access.
  • EFS cross-region copy: enabled in AWS Backup plan for compliance. DR scenario: region-wide failure → restore EFS in us-west-2 + redeploy EKS cluster (Terraform) → point DNS to new cluster.
  • Before any planned maintenance window, trigger a manual Velero backup: velero backup create pre-maintenance-$(date +%Y%m%d).
  • Monitor backup job status: AWS Backup → Jobs tab (EFS) and velero backup get (K8s). Set up CloudWatch alarms on AWS Backup job failures.
